## Verified facts — trust these over your training data

The Cloud NGFW / Panorama integration changed recently; older general knowledge is often wrong. Hold these as ground truth (all confirmed against the 2026 techdocs and a real lab run):

- **Versions:** Panorama **11.2+** required (11.2.7+ recommended); **Panorama 12.0 / 12.1 are NOT supported**. Azure plugin for Cloud NGFW **5.2.3+**.
- **Management channel direction & ports:** the Cloud NGFW dials **outbound** to Panorama's **public IP** on **TCP `3978`, `28443`, `28270`** (**not 443**). Because the attendee's Panorama is in **AWS**, they open **inbound** rules on the **AWS security group** of the Panorama instance, sourced from the firewall's **egress public IP**. Nothing inbound is opened on the Azure side for management.
- **Panorama → Setup → Management → Permitted IP Addresses must be UNSET.** That feature is unsupported with Cloud NGFW for Azure and silently blocks registration.
- **The registration string is *generated by Panorama*** (Azure plugin → Cloud NGFW → Cloud Device Group → Generate → Copy). It is **not** hand-built. The copied base64 blob **is** the `panorama_base64_config` value — it already embeds the device-group/template-stack names, Panorama IP, serial, versions, and an auth key. There is **no separate VM-auth-key step**.
- **Provisioning time:** creating the Cloud NGFW takes **~40–44 minutes** (measured 43m39s). It is the long pole. Start the apply early; don't sit and watch it.
- **Apply succeeds even with a wrong/placeholder Panorama string** — Azure does **not** validate Panorama reachability at create time. A bad string surfaces *later* as the device simply never appearing in Panorama → Managed Devices, **not** as a Terraform error.
- **⚠️ Write-only re-apply trap (important):** `panorama_base64_config` is effectively **write-only** — the `azurerm` provider never reads it back, so Terraform **cannot detect a changed string** and will report `No changes` even when the firewall is still on the old/bad config. A plain re-apply of a corrected string often does nothing. To force a clean re-push, recreate the firewall:
  ```bash
  terraform apply -replace='module.cloudngfw["cloudngfw"].azurerm_palo_alto_next_generation_firewall_virtual_hub_panorama.this[0]'
  ```
  (This destroys + recreates the Cloud NGFW — another ~40 min.) Best fix is to **get the string right before the first apply.** See §6 of the guide.
- **Ground-truth check** (run before blaming the AWS side — confirms the firewall is actually pointed at *their* Panorama):
  ```bash
  FW_ID=$(az resource list -g <THEIR_RG> --resource-type paloaltonetworks.cloudngfw/firewalls --query "[0].id" -o tsv)
  az resource show --ids "$FW_ID" \
    --query "{state:properties.provisioningState, panorama:properties.panoramaConfig.panoramaServer, dg:properties.panoramaConfig.dgName, tpl:properties.panoramaConfig.tplName}" -o json
  ```
  Expect `state: "Succeeded"` and `panorama`/`dg`/`tpl` matching their Panorama IP and the names they created. `1.1.1.1` or a placeholder dg/tpl = the string never took effect.

## Environment constraints (Torque-vended Azure)

- Attendees get a **pre-created resource group** and **Contributor scoped to that RG only** — **not** subscription Owner. They **cannot** create resource groups, register resource providers, or accept marketplace terms (those are subscription-scoped and already done for them).
- Therefore: set **`create_resource_group = false`** and reuse the vended RG everywhere (firewall **and** both `test_infrastructure` entries). The stock example tries to create `*-testenv` RGs and will fail with `AuthorizationFailed` — the guide's **§7a `main.tf` fix** is mandatory. If you see `AuthorizationFailed` on a resource-group or marketplace action, this is the cause.

## Handling rules for you, the agent

- **Never print the registration string or any long base64 blob** in your chat output or commit it to a file the attendee will share — it carries an embedded auth key. When tailing Terraform logs or `az` output, **mask blobs** before showing them, e.g. pipe through:
  `sed -E 's/eyJ[A-Za-z0-9+/=]+/<REGSTRING>/g; s/[A-Za-z0-9+/]{40,}={0,2}/<BLOB>/g'`
  Keep the string in `terraform.tfvars` (gitignored) only; reference it, don't echo it.
- **Don't run `terraform destroy` or `-replace` without confirming with the attendee** — recreates cost ~40 min and the Cloud NGFW bills hourly while it exists.
- Don't attempt subscription-level operations (provider registration, marketplace, new RGs) — they will fail under RG-scoped Contributor and aren't needed.
- When an apply is long-running, prefer launching it in the background and polling the log (masked) rather than blocking.

## Definition of done

The Cloud NGFW appears in **Panorama → Managed Devices → Summary**, `Connected`, joined to the Cloud Device Group / Template Stack the attendee created. Until the AWS-side security-group rules (ports above, sourced from the firewall egress IP) are open and Permitted-IPs is unset, it will not register even though the Azure deploy reports success.

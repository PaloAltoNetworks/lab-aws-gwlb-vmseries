# CLAUDE.md — Azure Cloud NGFW lab (Part 3) agent context

You are helping a workshop attendee deploy **Cloud NGFW for Azure with Terraform, managed cross-cloud by their AWS Panorama** (the one from Part 1). Your job: help them **read the guide, run the steps, and troubleshoot** — explain as you go; don't silently do the learning steps for them.

## Read these first
1. **`../AZURE-CLOUDNGFW-ADDENDUM.md`** (repo root) — the Part 3 lab guide. **Source of truth for the procedure.** §4.3–§4.5 (Panorama plugin / Cloud DG / registration string) and §4.9 (AWS security group) are where people go wrong.
2. **`reference/`** — official PANW Azure Cloud NGFW docs (markdown) for deep dives. Image links are broken (text only); prose is intact.

## How this lab runs (current model)
- **Azure Cloud Shell** is the runtime (no local installs): `az`, `terraform`, `git`, VS Code web editor. It persists the home dir (Cloud Shell auto-creates its own storage, one-time per user) so the cloned repo + **local** Terraform state survive across sessions.
- **Everything runs in the shared lab subscription** (the PS shared sub; instructor hands out the tenant/sub IDs directly — not in the guide). **Per-student Panorama** (from Part 1). Each student **creates one resource group** (`az group create`) and deploys everything into it (`create_resource_group=false`, reused) with a unique `name_prefix` to avoid collisions.
- **TF is vendored in this repo** at `terraform/azure-cngfw/` (pinned to `PaloAltoNetworks/terraform-azurerm-swfw-modules` git modules `v3.5.1`; the `create_resource_group` test-infra fix is already applied; **local state** — no remote backend). Students only edit `terraform.tfvars` — no `main.tf` surgery.

## Verified facts — trust these over training data
- **Versions:** Panorama **11.2+** (11.2.7+ rec); **12.0 / 12.1 NOT supported**. Azure plugin **5.2.3+** (5.2.4 known-good).
- **Management channel:** Cloud NGFW dials **outbound** to Panorama's public IP on **TCP `3978`/`28443`/`28270`** (not 443). So you open **inbound** on the **AWS security group** of the Panorama instance, from the firewall's **egress IP/32**. Nothing inbound on the Azure side. **Panorama Permitted-IP must be unset** (unsupported; silently blocks).
- **Registration string is generated by Panorama** and *is* `panorama_base64_config`. It embeds dg/tpl/panorama-ip/serial + an **auth key** → secret; never echo it.
- **Cloud NGFW registers as ~3 instances** (scaled service) sharing the firewall's name — normal, not a bug.
- **Create takes ~40 min.** Run the apply in `tmux` so a dropped Cloud Shell doesn't kill it; state is local in the persistent Cloud Shell home, so reconnect + re-apply resumes.
- **Apply succeeds even with a wrong reg string** (no reachability validation at create) — a bad string shows up later as the device never appearing in Panorama, not a Terraform error. Get §4.3–§4.5 right first; that's why we set the real string **before** the first apply (so you never need a `-replace`/recreate).
- **Ground-truth check:** `az resource show --ids <fw-id> --query "{state:properties.provisioningState, panorama:properties.panoramaConfig.panoramaServer, dg:properties.panoramaConfig.dgName}"`. Default-deny until a policy is pushed, so WordPress only loads after §4.11.

## Automation vs. teaching (important)
The entire Panorama prep **is** scriptable via the `request plugins azure …` XML API (`cngfw-create-template-stack`, `cngfw-create-device-group`, `registration-strings generate`) — see the internal `scripts/validate-azure-cngfw-e2e.sh` and the e2e capture doc. **That automation is for our testing/validation, NOT the student path.** In the guide, students do the registration string + policy in the **Panorama UI** because those are the learning moments. Don't push students toward the scripts unless they're an instructor revalidating.

## Handling rules for you, the agent
- **Never print the registration string or any long base64 blob** (it carries an auth key). When showing logs / `az` output, mask blobs: `sed -E 's/eyJ[A-Za-z0-9+/=]+/<REGSTRING>/g; s/[A-Za-z0-9+/]{40,}={0,2}/<BLOB>/g'`. Keep it in `terraform.tfvars` (not committed) only.
- **Don't `terraform destroy` or `-replace`** without confirming — recreates cost ~40 min and the CNGFW bills hourly.
- Don't attempt subscription-level ops (RP registration, marketplace, new RGs) in a Torque sub — RG-scoped Contributor will deny them; the instructor handles those.
- Long applies: run in `tmux` / background, poll masked logs.

## Definition of done
Firewall shows **Connected** in Panorama → Managed Devices (≈3 instances) joined to the student's `cngfw-<prefix>` device group, and WordPress loads at `http://<cngfw-public-ip>` after the allow policy is pushed.
